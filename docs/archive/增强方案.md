# local-mcp-search 增强方案

## 1. 背景

当前项目已经实现一个本地 `STDIO MCP Server`，用于给 `Codex`、`Claude Code` 提供本地代码库和知识库检索能力。

现有工具：

- `repo_overview`
- `index_status`
- `reindex`
- `code_exact_search`
- `symbol_search`
- `code_semantic_search`
- `kb_search`
- `open_spans`

现有底座：

- 本地 embedding 模型
- `LanceDB` 本地向量库
- `ripgrep` 精确检索
- Git / manifest 增量索引
- 可选后台自动增量更新

新增条件：

- 本机已经部署 reranker 库

最终目标：

- 让 `Codex` / `Claude Code` 少读无关文件
- 让模型少做大范围代码扫描
- 让 expensive model token 主要花在判断和修改上，而不是花在寻找上下文上

## 2. 总体判断

最值得增强的方向不是“继续增加更多搜索入口”，而是增加三类能力：

1. 搜索结果重排
2. 本地上下文压缩
3. 面向任务的聚合工具

原因：

- embedding 负责召回，但召回结果经常“相关但不够准”
- reranker 能显著减少误召回，让 top 3 更可靠
- 本地 MCP 可以先做筛选、排序、摘要、分组，再把少量高价值上下文交给模型
- 对 `Codex` / `Claude Code` 来说，最省 token 的工具通常不是返回更多内容，而是返回更少但更准的内容

## 3. 增强优先级

建议按以下顺序推进：

1. `rerank_search_results`
2. `code_context_pack`
3. `symbol_context`
4. `impact_analysis`
5. `doc_answer_context`
6. `file_outline`
7. `callsite_search`
8. `change_context`
9. `dependency_overview`
10. `session_cache`

其中第一阶段最建议先做：

- reranker 接入
- 上下文打包
- 符号上下文聚合
- 文件 outline

这四个工具对减少 token 最直接。

## 4. Reranker 接入

### 4.1 新工具：`rerank_search_results`

用途：

- 对已有搜索结果进行本地重排
- 输入 query 和候选片段
- 输出 rerank 后的 top-k

适用场景：

- `code_semantic_search` 返回多个相关候选，但排序不稳定
- `kb_search` 找到一批文档片段，需要挑最能回答问题的几个
- 混合检索之后，需要合并 exact、semantic、kb 的结果

建议输入：

```json
{
  "query": "工具调用结果如何写入历史消息",
  "candidates": [
    {
      "path": "src/agent.py",
      "line_start": 420,
      "line_end": 460,
      "text": "..."
    }
  ],
  "top_k": 5
}
```

建议输出：

```json
{
  "results": [
    {
      "path": "src/agent.py",
      "line_start": 424,
      "line_end": 460,
      "rerank_score": 0.92,
      "reason": "directly describes history message append behavior"
    }
  ]
}
```

实现建议：

- 新建 `reranker_client.py`
- 使用环境变量配置：
  - `RERANKER_BASE_URL`
  - `RERANKER_MODEL`
  - `RERANKER_API_KEY`
- 支持 OpenAI 兼容、Jina 兼容、FlagEmbedding 本地服务三种适配方式
- 第一版只支持一种本机接口也可以，但接口层要隔离

### 4.2 内置到现有搜索

不建议只暴露独立 `rerank_search_results`。更好的方式是：

- `code_semantic_search` 增加 `rerank: true`
- `kb_search` 增加 `rerank: true`
- 初召回 top 30
- rerank 后只返回 top 5

这样对 `Codex` / `Claude Code` 更自然，因为模型不用多调用一步。

建议默认：

- `code_semantic_search`: 初召回 30，rerank 返回 6
- `kb_search`: 初召回 20，rerank 返回 5

## 5. 上下文打包

### 5.1 新工具：`code_context_pack`

用途：

- 根据用户问题，在本地完成“检索 + rerank + 去重 + 截断 + 上下文打包”
- 一次返回可直接用于分析或修改的紧凑上下文包

这是最有 token 价值的工具。

适用场景：

- “帮我理解工具调用结果是怎么记录的”
- “我要修改某个功能，先找相关上下文”
- “查一下这个流程相关的关键代码和文档”

建议流程：

1. exact search
2. semantic search
3. kb search
4. rerank
5. 按文件分组
6. 合并相邻行号
7. 限制总字符数
8. 返回最小上下文包

建议输入：

```json
{
  "query": "工具调用结果写入历史消息的完整链路",
  "include_docs": true,
  "max_files": 5,
  "max_chars": 12000
}
```

建议输出：

```json
{
  "summary": "Found likely implementation in src/agent.py and related design notes in 工具系统建设.md.",
  "items": [
    {
      "path": "src/agent.py",
      "line_start": 424,
      "line_end": 460,
      "why_included": "history append implementation",
      "content": "..."
    }
  ],
  "omitted": [
    {
      "path": "src/tools.py",
      "reason": "lower rerank score"
    }
  ]
}
```

token 控制规则：

- 默认 `max_chars = 12000`
- 单文件最多 2 个 span
- 相邻 span 自动合并
- 相同 chunk 不重复返回
- 文档和代码分开标注

### 5.2 与 `open_spans` 的关系

`open_spans` 是低层工具。

`code_context_pack` 是高层工具。

建议：

- 精确任务用 `open_spans`
- 模糊任务用 `code_context_pack`
- `code_context_pack` 内部可以复用 `open_spans`

## 6. 符号上下文

### 6.1 新工具：`symbol_context`

用途：

- 输入一个 symbol，返回定义、附近上下文、调用点、相关文档

适用场景：

- “解释 `execute_tool` 是怎么工作的”
- “修改 `build_graph` 前先看看相关上下文”
- “这个类在哪里定义、哪里用到”

建议流程：

1. `symbol_search` 找定义
2. `code_exact_search` 找引用
3. 对引用片段 rerank
4. 打开定义上下文
5. 返回定义 + top callsites

建议输入：

```json
{
  "symbol": "execute_tool",
  "include_callers": true,
  "include_docs": false,
  "max_callers": 8
}
```

建议输出：

```json
{
  "definition": {
    "path": "src/tools.py",
    "line_start": 561,
    "line_end": 625,
    "content": "..."
  },
  "callers": [
    {
      "path": "src/agent.py",
      "line_start": 2517,
      "line_end": 2560,
      "why_included": "executes planned tool call"
    }
  ]
}
```

token 收益：

- 避免模型先查定义，再查引用，再打开多个文件
- 一次返回“最小必要上下文”

## 7. 文件结构摘要

### 7.1 新工具：`file_outline`

用途：

- 返回单个文件的结构化 outline，而不是全文

适用场景：

- 大文件阅读前
- 不确定要打开哪段代码
- 让模型先看函数列表、类列表、主要区域

建议输入：

```json
{
  "path": "src/agent.py",
  "include_private": true
}
```

建议输出：

```json
{
  "path": "src/agent.py",
  "language": "python",
  "items": [
    {
      "kind": "class",
      "name": "MissionManager",
      "line_start": 156,
      "line_end": 344
    },
    {
      "kind": "function",
      "name": "execute_tool",
      "line_start": 561,
      "line_end": 625
    }
  ]
}
```

实现建议：

- 第一版用正则支持 Python / TypeScript / JavaScript
- 第二版接入 Tree-sitter

token 收益：

- 大文件不再直接展开
- 模型可以按 outline 精确请求 span

## 8. 调用点检索

### 8.1 新工具：`callsite_search`

用途：

- 查某个 symbol 的调用点，并返回调用上下文

与 `code_exact_search` 的区别：

- `code_exact_search` 查所有文本出现
- `callsite_search` 尽量只返回调用点或使用点

适用场景：

- “哪些地方调用了 `model_chat`”
- “改 `execute_tool` 会影响哪里”
- “这个配置项在哪里实际读取”

建议输入：

```json
{
  "symbol": "model_chat",
  "max_results": 10,
  "rerank_query": "runtime execution path"
}
```

建议输出：

```json
{
  "results": [
    {
      "path": "src/agent.py",
      "line_start": 1415,
      "line_end": 1435,
      "snippet": "planner_chat(...) calls model_chat(...)",
      "score": 0.88
    }
  ]
}
```

## 9. 影响分析

### 9.1 新工具：`impact_analysis`

用途：

- 输入文件、symbol 或自然语言修改意图，返回可能受影响的代码区域

适用场景：

- “我要改工具执行结果格式，会影响哪些地方”
- “修改 history message 结构前，先找相关链路”
- “这个函数签名要改，相关调用有哪些”

建议流程：

1. 根据输入判断是 path、symbol 还是自然语言
2. 找定义和引用
3. 找相关测试
4. 找相关文档
5. rerank
6. 返回受影响区域清单

建议输出：

```json
{
  "impact_items": [
    {
      "path": "src/tools.py",
      "line_start": 561,
      "line_end": 625,
      "kind": "definition",
      "risk": "high",
      "reason": "core tool execution API"
    },
    {
      "path": "tests/test_tools.py",
      "line_start": 1,
      "line_end": 80,
      "kind": "test",
      "risk": "medium",
      "reason": "likely validates tool execution behavior"
    }
  ]
}
```

token 收益：

- 模型不用自己全局探索
- 修改前能快速获得关键影响面

## 10. 文档问答上下文

### 10.1 新工具：`doc_answer_context`

用途：

- 面向“为什么这样设计”“有没有方案文档”这类问题，返回最相关文档片段和引用来源

与 `kb_search` 的区别：

- `kb_search` 返回候选
- `doc_answer_context` 返回可直接回答问题的压缩文档上下文

建议流程：

1. `kb_search` 初召回
2. rerank
3. 合并同一文档相邻章节
4. 提取标题链
5. 返回 top 3 文档片段

建议输入：

```json
{
  "question": "为什么工具结果要写入 messages.jsonl",
  "max_docs": 3,
  "max_chars": 8000
}
```

token 收益：

- 文档类问题不需要打开多篇 markdown
- 模型可基于短上下文回答

## 11. 变更上下文

### 11.1 新工具：`change_context`

用途：

- 返回当前 Git 变更相关的最小上下文

适用场景：

- 代码 review
- 总结当前改动
- 继续上次未完成工作
- 生成 commit message

建议输入：

```json
{
  "include_untracked": true,
  "include_docs": false,
  "max_chars": 12000
}
```

建议输出：

```json
{
  "changed_files": [
    {
      "path": "src/agent.py",
      "status": "modified",
      "summary": "modified model output validation path"
    }
  ],
  "context": [
    {
      "path": "src/agent.py",
      "line_start": 1200,
      "line_end": 1280,
      "content": "..."
    }
  ]
}
```

实现建议：

- 使用 `git diff --name-only`
- 使用 `git diff --unified=3`
- 对 diff 太大的文件只返回 hunk summary
- 可结合 reranker 判断哪些 hunk 与用户问题最相关

token 收益：

- review / commit / resume 场景下非常省 token
- 避免模型读取大量未变更代码

## 12. 依赖和结构总览

### 12.1 新工具：`dependency_overview`

用途：

- 返回项目依赖、入口、配置文件、测试框架、构建方式

适用场景：

- 新项目第一次打开
- 修改前判断技术栈
- 模型决定运行什么测试命令

建议输出：

```json
{
  "language": ["python"],
  "package_files": ["pyproject.toml", "requirements.txt"],
  "test_dirs": ["tests"],
  "entrypoints": ["myAgent.py"],
  "likely_commands": {
    "test": "pytest",
    "run": "python myAgent.py"
  }
}
```

token 收益：

- 减少模型打开多个配置文件
- 更快决定测试和运行命令

## 13. 会话级缓存

### 13.1 新工具：`session_cache`

用途：

- 缓存最近查询、已打开 span、已确认关键文件

适用场景：

- 同一个任务中多轮问答
- 模型重复搜索同一问题
- 防止重复返回同一大段上下文

建议能力：

- `cache_put`
- `cache_get`
- `cache_list`
- `cache_clear`

但不建议一开始暴露四个工具。

更好的方式：

- 在 MCP server 内部自动维护短期缓存
- 搜索工具返回 `cache_key`
- `open_spans` 如果重复打开同一 span，可返回 “already_opened” 提示或更短内容

token 收益：

- 减少同一任务内的重复上下文消耗

## 14. Claude Code Resources 增强

Claude Code 对 MCP resources 支持更完整。可以新增资源：

- `repo://overview`
- `repo://file/<path>`
- `repo://symbol/<symbol>`
- `kb://doc/<doc_id>`
- `kb://section/<doc_id>#<heading>`
- `git://changes`

价值：

- 用户可以直接引用资源
- Claude Code 可以把常用上下文作为稳定入口
- 对文档和符号尤其有用

优先级：

- 在工具能力稳定后实现
- 不建议早于 reranker 和 context pack

## 15. 推荐实施路线

### 第一阶段：高收益低风险

目标：

- 利用本机 reranker 提高 top-k 质量
- 减少模型打开错误文件的概率

任务：

1. 新增 `reranker_client.py`
2. 给 `code_semantic_search` 增加 rerank
3. 给 `kb_search` 增加 rerank
4. 新增 `file_outline`
5. 新增 `code_context_pack`

预期收益：

- 同样问题下，返回片段更少
- top 3 更准
- `open_spans` 调用次数减少

### 第二阶段：任务聚合

目标：

- 把常见多步探索压成本地一次工具调用

任务：

1. 新增 `symbol_context`
2. 新增 `callsite_search`
3. 新增 `impact_analysis`
4. 新增 `doc_answer_context`

预期收益：

- “查定义 + 查引用 + 打开片段”的链路本地化
- 修改前影响面分析更快

### 第三阶段：工作流增强

目标：

- 支持 review、resume、commit、跨轮任务

任务：

1. 新增 `change_context`
2. 新增内部 session cache
3. 新增 Claude Code resources
4. watcher 从轮询升级到文件事件监听

预期收益：

- review / commit 场景显著减少上下文
- 重复搜索减少

## 16. 建议第一批实现的工具定义

最建议优先实现以下 4 个：

### 16.1 `file_outline`

描述：

> Return a structure-only outline of a source file, including classes, functions, methods, constants, and line ranges. Use this before opening large files.

### 16.2 `code_context_pack`

描述：

> Build a compact task-specific context pack by combining exact search, semantic search, optional docs search, reranking, deduplication, and span merging. Use this when the user asks a broad implementation question and you need enough context without opening many files.

### 16.3 `symbol_context`

描述：

> Return the definition, nearby implementation context, and most relevant callsites for a named symbol. Use this before modifying or explaining a specific function, class, or exported API.

### 16.4 `change_context`

描述：

> Return a compact summary and focused context for current Git changes. Use this for reviews, commit messages, resuming work, or understanding what changed without reading entire files.

## 17. Reranker 对 token 成本的具体价值

reranker 本身不直接减少 token。它通过减少错误上下文来间接省 token。

典型变化：

- 无 rerank：semantic top 8 返回 8 个片段，模型再打开 3 到 5 个文件
- 有 rerank：初召回 30，本地重排后只返回 top 3 到 top 5
- 结果：远端模型少读 30% 到 70% 的无关片段

最重要的实践：

- 不要把 reranker 分数低的内容返回给模型
- 不要返回 reranker 的全部候选
- 返回 `why_included`，让模型知道为什么这段值得看

## 18. 风险与约束

### 18.1 工具过多导致模型选择困难

对策：

- 高层工具少而明确
- 低层工具保留但描述清晰
- 推荐默认路径：
  - 模糊任务：`code_context_pack`
  - 具体符号：`symbol_context`
  - 文档问题：`doc_answer_context`
  - review：`change_context`

### 18.2 本地 reranker 延迟

对策：

- 初召回限制在 20 到 50
- rerank 文本截断到 1k 到 2k 字符
- 对相同 query + chunk_id 做缓存

### 18.3 上下文压缩误删关键信息

对策：

- 返回 `omitted` 列表
- 提供 `open_spans` 继续展开
- 对定义、调用点、测试优先保留

### 18.4 索引和工作区变化不同步

对策：

- `index_status` 继续暴露 freshness 信息
- `code_context_pack` 开始前可检查索引状态
- 变更文件优先走实时文件读取，而不是完全依赖旧索引

## 19. 最终建议

下一步不要先扩大量底层搜索工具，而应先实现：

1. reranker 接入
2. `file_outline`
3. `code_context_pack`
4. `symbol_context`

这四项能把当前系统从“检索工具集合”升级成“上下文预算控制器”。

对 `Codex` 和 `Claude Code` 来说，这才是最直接的价值：本地模型负责找、排、压缩，远端昂贵模型只负责理解、决策和修改。

# 安装部署程序方案

## 1. 目标

本方案面向本项目的商业化分发，目标是在用户机器环境差异较大的前提下，仍然能稳定完成安装、配置、启动、验证和后续修复。

这里的“setup 程序”不只是一个安装器界面，而是一套完整的部署机制，负责：

- 安装本项目运行时
- 检测用户机器环境
- 配置本地 embedding 后端
- 判断当前机器应进入哪一种运行模式
- 注册 `Codex` / `Claude Code` 的 MCP 配置
- 在安装结束时给出可验证的健康状态

本方案优先追求可交付和可维护，不追求一步到位解决所有平台和所有推理后端。

## 2. 当前项目现状

从当前仓库实现看，项目已经具备本地可用闭环，但还没有达到“可直接商业安装”的状态。

当前特点：

- 运行平台明显偏 `Windows + PowerShell`
- 主流程依赖本地 OpenAI 兼容 `embedding` 接口
- 当前默认后端是 `LM Studio`
- 默认模型为 `text-embedding-bge-base-zh`
- `reindex` 依赖 embedding 服务可用
- `reranker` 已支持不可用时自动回退
- `embedding` 不可用时，目前仍偏“硬失败”而不是“带降级启动”

当前明显不适合直接面向用户分发的点：

- [run-local-mcp-search.ps1](/D:/trae_prj/mcp_sd/run-local-mcp-search.ps1) 和 [use-local-mcp-search.ps1](/D:/trae_prj/mcp_sd/use-local-mcp-search.ps1) 默认使用开发机上的硬编码配置路径
- 启动脚本会先做 `reindex`，用户机器若 embedding 后端未准备好，安装流程容易被卡住
- 当前没有统一的“环境探测 / 健康检查 / 修复建议”命令
- 当前没有对“CPU 模式 / 无 CUDA / 节电模式”的正式产品策略定义

因此，商业化之前应先把“安装”与“运行模式判定”设计清楚。

## 3. 产品化原则

### 3.1 先支持单平台

第一版建议只支持：

- `Windows 10/11 x64`

原因：

- 当前脚本、接入方式、用户习惯都已经偏 Windows
- `Codex` / `Claude Code` 的本地 MCP 配置也已围绕 PowerShell 实现
- 如果第一版同时做 macOS / Linux，setup 和诊断复杂度会明显膨胀

### 3.2 先支持单一主后端

第一版建议主支持：

- `LM Studio` 作为本地 embedding 服务

理由：

- 当前项目已实际接入并验证
- 提供 OpenAI 兼容接口，改造成本最低
- 可兼容 GPU 和 CPU 两种执行环境

但要注意：

- 是否允许把 `LM Studio` 与模型文件一起打包分发，必须先做许可证与分发条款审查
- 如果许可证或体积不适合，第一版应采用“检测既有 LM Studio / 引导安装 / 接管配置”模式，而不是强行捆绑

### 3.3 优先支持降级，而不是硬失败

setup 程序应把用户机器划分为不同能力档，而不是只允许“完全成功”。

建议定义三种运行模式：

- `Full`
  - 本地 embedding 服务可用
  - 模型可正常返回向量
  - 允许完整语义检索与索引构建
- `Compatible`
  - embedding 服务可用，但性能不足，或仅 CPU 模式
  - 允许语义检索，但默认关闭自动重建，仅保留手动或增量重建
- `Basic`
  - embedding 服务不可用
  - 系统仍可启动，但仅开放精确检索、符号检索、概览、上下文打开等非语义能力

商业化时必须让用户看到自己当前属于哪个模式，而不是让其自行猜测。

## 4. 用户环境分类

setup 程序要处理的核心不是“有没有显卡”这么简单，而是以下几类环境差异：

- 没有独显
- 有 NVIDIA 显卡，但未安装驱动
- 有驱动，但 CUDA 运行条件不满足
- 有 GPU，但 `LM Studio` 当前未启用 GPU 推理
- 机器处于节电模式，导致性能显著下降
- `LM Studio` 已安装，但未启动本地 server
- `LM Studio` server 已启动，但模型未加载
- 模型已加载，但 API key / 端口 / base URL 不匹配
- embedding 模型存在，但不是本项目要求的维度或名称
- 用户机器无 Python、无 `ripgrep`、无 `git`

其中，真正影响本项目可用性的关键探针是：

- OpenAI 兼容接口是否可连通
- `/models` 能否看到目标模型
- `/embeddings` 是否能成功返回向量
- 返回维度是否与索引维度一致
- 单次 embedding 延迟是否在可接受范围内

也就是说，GPU/CUDA 只是“原因”，setup 真正要判断的是“当前后端能力是否达标”。

## 5. V1 安装方案范围

第一版建议把 scope 控制在如下范围：

- 安装本项目运行文件
- 安装或解压内置 Python 运行时
- 注册本项目命令行入口和启动器
- 检测并接入现有 `LM Studio`
- 生成项目配置文件
- 进行一次健康检查
- 按检测结果写入运行模式
- 可选注册 `Codex` / `Claude Code` MCP

第一版暂不建议在 setup 中承担以下职责：

- 自动安装 GPU 驱动
- 自动修复 CUDA 环境
- 自动下载超大模型文件
- 在后台长期托管和管理多种模型后端
- 一次性支持多个 embedding 后端品牌

这些能力可以进入后续版本，但不适合成为首版交付门槛。

## 6. 推荐交付形态

### 6.1 安装器类型

第一版建议采用：

- `Inno Setup` 或 `WiX` 打包 `Windows Installer`

更偏实用的选择是：

- `Inno Setup` 负责安装界面、文件拷贝、快捷方式、卸载
- 本项目自带一个 `setup-bootstrap.ps1` 或 `setup-bootstrap.exe` 执行真正的探测、配置和验证逻辑

原因：

- 纯安装器脚本不适合承载复杂环境检测逻辑
- 项目本身已有 PowerShell 基础，迁移成本低
- 后续也可把 bootstrap 逐步收敛成 Python CLI 或独立小型 exe

### 6.2 运行时打包

第一版建议不要依赖用户本机自备 Python。

建议打包：

- 内置 Python runtime
- 本项目 wheel 或冻结后的可执行入口
- `ripgrep`
- 默认配置模板

可选不打包：

- `git`

原因：

- 本项目在没有 `git` 时仍可退化到 manifest 变更检测
- 但没有 Python 时，用户环境不确定性太高

## 7. setup 程序职责拆分

建议把 setup 流程拆成五个阶段。

### 阶段 A：基础环境探测

检测项：

- 操作系统版本
- `x64` 架构
- 可写目录权限
- 是否已有旧版本安装
- 是否已有用户级配置

输出：

- 安装目标目录
- 配置目录
- 日志目录

建议目录：

- 程序目录：`%ProgramFiles%\YourProduct\`
- 机器级配置：`%ProgramData%\YourProduct\`
- 用户级配置：`%LocalAppData%\YourProduct\`
- 日志目录：`%LocalAppData%\YourProduct\logs\`

### 阶段 B：推理后端探测

检测项：

- `LM Studio` 是否安装
- `LM Studio` 是否正在运行
- 本地 server 端口是否可达
- API key 是否存在
- `/models` 是否可访问
- `text-embedding-bge-base-zh` 是否存在
- `/embeddings` 调用是否成功
- 向量维度是否为 `768`
- 近 1 到 3 次请求平均延迟

如果需要进一步增强，可附加检测：

- Windows 电源计划是否为节能
- NVIDIA 驱动是否存在
- GPU 是否可见

但这些附加项用于辅助诊断，不作为唯一准入条件。

### 阶段 C：能力分级

根据探测结果写入运行模式：

- `Full`
- `Compatible`
- `Basic`

推荐判定逻辑：

- `Full`
  - `/embeddings` 成功
  - 模型存在
  - 维度正确
  - 延迟低于设定阈值
- `Compatible`
  - `/embeddings` 成功
  - 但延迟偏高，或检测到 CPU 模式
- `Basic`
  - `/embeddings` 失败，或模型不存在

推荐阈值：

- `< 500ms`：可视为正常
- `500ms - 3000ms`：可视为兼容但降级
- `> 3000ms` 或失败：不可用于安装期重建

### 阶段 D：配置落地

setup 应生成统一配置，而不是继续依赖开发期 JSON 路径。

建议生成：

- `app-config.json`
- `embedding-config.json`
- `runtime-state.json`

至少包含：

- `base_url`
- `api_key`
- `model_name`
- `workspace_root`
- `index_dir`
- `retrieval_mode`
- `auto_reindex_enabled`
- `last_healthcheck_at`
- `last_healthcheck_result`

### 阶段 E：安装后验证

安装结束前必须做一次正式验证，并出结果页。

最少验证项：

- 程序入口能启动
- `index_status` 可返回
- exact search 可工作
- 如 embedding 可用，则 `/embeddings` 测试成功
- 如为 `Full` / `Compatible` 模式，则执行一个极小规模试探性索引

安装成功页必须给出：

- 当前运行模式
- embedding 状态
- 是否启用语义检索
- 是否启用自动重建
- 配置文件位置
- 日志文件位置

## 8. 推荐运行模式策略

### 8.1 Full 模式

条件：

- embedding 后端可用且性能达标

行为：

- 开启完整语义检索
- 允许 `reindex full / incremental / auto`
- 可开启自动增量重建

### 8.2 Compatible 模式

条件：

- embedding 后端可用，但性能明显偏慢
- 常见原因：CPU 推理、节电模式、共享显存、低功耗本

行为：

- 允许语义检索
- 安装阶段不做全量索引，最多只做小样本验证
- 默认关闭后台自动重建
- 首次真正全量索引改为用户手动触发

这是商业化时最关键的折中模式。  
它保证“能用”，但不假装“和高性能机器一样体验”。

### 8.3 Basic 模式

条件：

- embedding 后端不可用

行为：

- 仍允许启动 MCP server
- 仅开放不依赖 embedding 的能力
- 禁止 `reindex`
- `code_semantic_search` / `kb_search` 返回明确错误：当前环境未启用语义检索

这能显著降低安装失败率，也更符合真实用户环境。

## 9. setup 程序界面流程建议

建议安装界面按以下顺序组织：

1. 欢迎页
2. 许可证与隐私说明
3. 安装目录选择
4. 运行环境检测
5. 本地 embedding 后端接入
6. MCP 客户端注册选项
7. 安装执行
8. 健康检查
9. 完成页

“运行环境检测”页应明确显示：

- `LM Studio`：已安装 / 未安装
- 本地 API：可连接 / 不可连接
- 目标模型：已找到 / 未找到
- embedding 调用：成功 / 失败
- 当前模式：`Full` / `Compatible` / `Basic`

如果检测到问题，setup 应给出具体动作，不要只报错：

- 去启动 `LM Studio`
- 去加载指定模型
- 去修改端口或 API key
- 切换到 `Basic` 模式继续安装

## 10. 对当前代码的必要改造

在真正做 setup 程序之前，建议先完成下面这些代码级改造，否则安装器只能包住问题，不能解决问题。

### 10.1 去掉开发机硬编码路径

当前脚本中默认路径直接指向开发机：

- `C:\Users\yyyx\Documents\models-setting\my-embd-bge-zh.json`
- `C:\Users\yyyx\Documents\models-setting\qwen3-reranker_lingya.json`

商业版必须改成：

- 从统一配置目录读取
- 若配置不存在，则由 setup 写入

### 10.2 把 embedding 初始化改成可降级

当前 `RetrievalService` 初始化时直接创建 `EmbeddingClient`。  
这会导致 embedding 后端有问题时，整个服务偏向启动失败。

建议改成：

- 启动时先做惰性探测
- 将 embedding 能力标记到 service state
- 非语义工具继续可用

### 10.3 增加 `doctor` 命令

建议新增：

- `python -m local_mcp_search.cli doctor`

输出机器可读 JSON，供 setup 程序直接消费。

建议包含：

- app version
- config path
- backend type
- embedding reachable
- model present
- embedding dimensions
- embedding latency
- gpu visible
- power mode
- recommended retrieval mode
- actionable fixes

### 10.4 增加显式运行模式

建议把运行模式写成正式配置项：

- `retrieval_mode = full | compatible | basic`

而不是通过零散开关间接推导。

### 10.5 启动脚本停止默认先 `reindex`

对于用户机器，setup 或首次启动不应默认先做完整 `reindex`。

建议改成：

- 先 `doctor`
- 根据模式决定是否允许索引
- `Full` 才默认允许首次全量索引
- `Compatible` 仅提示用户稍后执行
- `Basic` 完全跳过

## 11. 推荐配置与状态文件

建议引入统一配置目录结构：

```text
%ProgramData%\YourProduct\
  app-config.json
  embedding-config.json
  runtime-state.json

%LocalAppData%\YourProduct\logs\
  setup.log
  app.log
  doctor.log
```

其中：

- `app-config.json` 存产品级稳定配置
- `embedding-config.json` 存本地推理后端配置
- `runtime-state.json` 存最近一次检测结果与运行模式

## 12. 错误处理策略

商业安装流程里最忌讳“失败原因不透明”。

建议所有失败都归类到有限错误码：

- `E1001` 未检测到本地 embedding 后端
- `E1002` embedding API 不可访问
- `E1003` API key 无效
- `E1004` 目标模型不存在
- `E1005` embedding 调用失败
- `E1006` 返回维度与预期不符
- `E1007` 当前环境仅支持 Basic 模式

setup 完成页或日志中应同时给出：

- 错误码
- 人类可读说明
- 建议修复动作

## 13. 版本路线建议

### V1

- 仅 Windows
- 接入现有 `LM Studio`
- 安装器完成环境检测、配置生成、模式分级、MCP 注册
- 不强制 bundling 模型

### V1.1

- 增加 `doctor` 与自修复入口
- 增加开机后首次运行引导
- 优化 CPU 模式体验

### V2

- 考虑支持第二种本地 embedding 后端
- 考虑托管模型下载
- 考虑支持企业离线安装包

## 14. 推荐结论

商业化第一步，不应先做“万能安装器”，而应先把系统能力分层：

- `Full`：完整语义能力
- `Compatible`：可用但降级
- `Basic`：不带语义能力，但服务仍能启动

随后再围绕这三种模式设计 setup 程序。

对当前项目，最务实的落地顺序是：

1. 先完成代码层的可降级改造和 `doctor` 命令
2. 再做 `Windows` 单平台安装器
3. 首版只接入 `LM Studio`
4. 安装器只负责探测、配置、验证和注册，不负责解决所有驱动问题

这样才能把部署复杂度控制在可交付范围内，同时让用户机器环境差异真正可管理。

# 功能详解

## 1. 文档范围

本文只描述本项目当前仓库中已经实现、已经可用、已经可以对外说明的能力，不包含仅在方案文档中提出但尚未落地的规划项。

本文重点回答两件事：

- 本项目已经如何增强 `Codex` 与 `Claude Code`
- 本项目已经如何降低远端大模型的上下文消耗与使用成本

本项目当前定位是一个本地 `STDIO MCP Server`，通过本地检索、索引、重排和上下文压缩，为 `Codex` / `Claude Code` 提供更适合真实代码库工作的检索能力。

## 2. 已实现的总体能力

当前已经实现并对外暴露的 MCP tools 为：

- `code_exact_search`
- `symbol_search`
- `code_semantic_search`
- `kb_search`
- `repo_overview`
- `code_context_pack`
- `file_outline`
- `symbol_context`
- `doc_answer_context`
- `change_context`
- `dependency_overview`
- `open_spans`
- `index_status`
- `reindex`

底层能力已经落地为：

- 本地 `ripgrep` 精确检索
- 本地 OpenAI 兼容 embedding 接口
- 本地 `LanceDB` 向量索引
- 可选 reranker 二阶段重排
- Git / manifest 驱动的增量索引
- 可选后台自动增量更新

这意味着本项目已经不是一个单纯的“向量搜一下”的小工具，而是一套面向代码助手工作流的本地上下文服务。

## 3. 对 Codex / Claude Code 的增强

### 3.1 统一的本地 MCP 检索入口

本项目已经把代码检索、文档检索、上下文回取、依赖概览、变更上下文等能力统一为一组 MCP tools，能被 `Codex` 与 `Claude Code` 直接调用。

这带来的增强是：

- 代码助手不需要先盲读整个仓库
- 可以先检索、再定位、再拉精确上下文
- 本地工具负责找候选，远端模型负责理解、决策与修改

对客户端来说，这种增强不是“多一个搜索按钮”，而是把原本分散的人类操作流程本地化、工具化了。

### 3.2 面向客户端习惯的工具拆分

本项目不是把所有能力堆进一个大而全的 `search`，而是已经按任务类型拆分成多种工具：

- 查具体字符串，用 `code_exact_search`
- 查定义位置，用 `symbol_search`
- 查相似实现，用 `code_semantic_search`
- 查文档方案，用 `kb_search`
- 拉精确片段，用 `open_spans`
- 取紧凑任务上下文，用 `code_context_pack`

这样的拆分对 `Codex` / `Claude Code` 有直接价值：

- 模型更容易选对工具
- 工具描述更容易写清楚
- 返回格式更稳定
- 可以避免“本该精确查找，却误用了语义搜索”的常见浪费

当前 [src/local_mcp_search/server.py](/D:/trae_prj/mcp_sd/src/local_mcp_search/server.py) 已经为各工具写了较细的 description，用来引导模型正确选路。

### 3.3 支持 Codex 与 Claude Code 的统一接入

项目已经提供脚本化接入方式，不需要用户手工逐条配置：

- 可注册 `Codex` 的 MCP
- 可注册 `Claude Code` 的 MCP
- 可为 `Claude Code` 生成项目级 `.mcp.json`
- 可按项目恢复最近会话

当前已实现的脚本层增强包括：

- [run-local-mcp-search.ps1](/D:/trae_prj/mcp_sd/run-local-mcp-search.ps1)
- [use-local-mcp-search.ps1](/D:/trae_prj/mcp_sd/use-local-mcp-search.ps1)
- [cpx.ps1](/D:/trae_prj/mcp_sd/cpx.ps1)

其中 `cpx` 已经把四件事情合并成一个统一入口：

- 识别当前 workspace root
- 刷新本地索引
- 更新 `local-search` MCP 配置
- 恢复最近的 `Codex` 或 `Claude Code` 会话

这对商业使用尤其重要，因为它减少了客户端切换和多项目切换时的人工配置成本。

### 3.4 面向会话恢复的增强

当前项目已经实现了面向工作连续性的包装式增强：

- 能按 workspace 恢复最近的 `Codex` session
- 能按 workspace 恢复最近的 `Claude Code` session
- 支持 `Fresh`、`Pick`、`Fork` 等恢复方式

这类增强不是检索算法上的创新，但对真实开发工作流价值很高。  
它让本地索引、MCP 配置和具体会话恢复绑定到同一个项目维度，减少“恢复到了错误会话”或“会话恢复了但 MCP 没注册”的问题。

## 4. 已实现的检索与上下文能力

### 4.1 `repo_overview`：项目总览

`repo_overview` 已实现一个低成本的仓库扫描器，返回：

- 顶层目录
- 顶层文件
- 常见扩展名统计
- 文档入口候选
- 可能的代码入口

它的价值在于：

- 初次进入项目时，先给模型一张便宜的地图
- 避免模型一上来就展开大文件
- 避免为判断项目结构而浪费大量上下文

### 4.2 `code_exact_search`：精确文本检索

`code_exact_search` 当前优先使用 `ripgrep` 的 JSON 输出进行精确匹配，找不到 `ripgrep` 时还会退化为 Python 逐文件扫描。

它适合：

- 查函数名
- 查类名
- 查错误文本
- 查配置键
- 查路由字符串
- 查环境变量名

它对成本控制很关键，因为很多任务本质上并不需要 embedding。  
如果用户问的是明确标识符，用精确搜索比语义检索更快、更准，也更省 token。

### 4.3 `symbol_search`：声明级定位

`symbol_search` 已实现基于多组声明模式的检索，覆盖：

- `def`
- `class`
- `function`
- `interface`
- `type`
- `enum`
- `const`
- 常见 `export` 声明

它的作用不是查“所有出现”，而是优先找“定义位置”。  
这能帮助 `Codex` / `Claude Code` 在改函数、改类、查 API 时少走弯路。

### 4.4 `code_semantic_search`：代码语义检索

`code_semantic_search` 已实现以下链路：

- 查询文本 embedding
- 在本地 `LanceDB` 中做向量检索
- 按代码类型和语言过滤
- 返回语义相似代码片段

同时结果里已保留：

- `score`
- `vector_score`
- `rerank_score`
- `why_matched`

这样做的意义是：

- 客户端可以拿到“粗召回分数”和“精排分数”
- 更容易判断结果是向量相似还是 reranker 精排后得到
- 便于后续调试召回质量

### 4.5 `kb_search`：知识库语义检索

`kb_search` 已面向 README、文档、ADR、工作记录等知识文件提供语义检索。

这项能力增强了 `Codex` / `Claude Code` 对“文档类问题”的处理能力，例如：

- 设计方案在哪里
- 当前部署说明是什么
- 某个架构决策写在哪

它避免模型为了回答一个文档问题去打开多篇 Markdown。

### 4.6 `open_spans`：精确上下文回取

`open_spans` 是当前实现中非常关键的低层工具。  
它只根据指定路径和行号范围返回精确片段，并且：

- 限制总字符数
- 校验路径不能逃出 workspace root
- 不允许无边界地展开整仓文件

这项能力是降本的核心基础设施之一。  
因为真正贵的不是“搜索”，而是把过多无关代码送进远端模型。

### 4.7 `code_context_pack`：任务级上下文打包

`code_context_pack` 已经落地，不是规划项。  
当前实现会在本地完成：

- 语义召回
- rerank
- 去重
- 相邻 span 合并
- 上下文扩边
- 字符预算裁剪

当前实现细节包括：

- 每个命中点默认向前后各扩 `8` 行
- 相同 `(path, line_start, line_end)` 结果会去重
- 同文件相邻或接近的 span 会合并
- 总字符数超过预算会截断

这项能力是当前系统中最直接的“上下文预算控制器”。  
对 `Codex` / `Claude Code` 来说，它把原本需要多次搜索、多次打开文件、多次人工筛选的流程压缩成一次本地工具调用。

### 4.8 `file_outline`：大文件结构摘要

`file_outline` 已实现对单文件的轻量 outline 提取，当前能识别：

- class
- function
- interface
- type
- const
- route 风格声明

输出包括：

- 文件路径
- 行数
- 结构项列表
- 每项的种类、名字、行号、预览文本

它的价值在于：

- 打开大文件前先知道里面有什么
- 让模型按结构去请求更小的 span
- 减少“一次性展开整个 1000 行文件”的冲动

### 4.9 `symbol_context`：围绕 symbol 的聚合上下文

`symbol_context` 已实现对 symbol 的聚合式取数：

- 用 `symbol_search` 找定义
- 用 `code_exact_search` 找引用
- 对结果去重
- 再调用 `build_context_pack` 组织紧凑上下文

这对 `Codex` / `Claude Code` 的提升很直接：

- 不需要模型自己先找定义再找引用
- 不需要模型手动判断哪些引用值得展开
- 对“解释某个函数”“修改某个类”这种任务非常适合

### 4.10 `doc_answer_context`：文档回答上下文

`doc_answer_context` 已实现“文档搜索 + 上下文压缩”组合能力：

- 先走 `kb_search`
- 再把命中文档片段打包成可直接回答问题的紧凑上下文

它比单纯 `kb_search` 更进一步，因为它返回的不是一组候选，而是更适合直接回答的压缩片段。

### 4.11 `change_context`：变更上下文

`change_context` 已面向 review、resume、理解本地改动场景落地。

当前实现会：

- 识别当前变更文件
- 对变更文件做 outline
- 读取一部分首段上下文
- 在总字符预算内返回紧凑结果

这项能力对 `Codex` / `Claude Code` 的增强体现在：

- 继续中断工作时更快进入状态
- review 时减少读取未变更文件
- 有助于生成变更总结和下一步修改计划

### 4.12 `dependency_overview`：依赖和构建总览

`dependency_overview` 已实现对常见依赖和构建文件的聚合读取，覆盖：

- `package.json`
- `pyproject.toml`
- `requirements.txt`
- `uv.lock`
- `Cargo.toml`
- `go.mod`
- `Dockerfile`
- `docker-compose.yml`

它能快速返回：

- 项目可能的技术栈
- 构建配置
- 测试和运行入口线索

这对初次进入陌生仓库的代码助手很有帮助，也能减少反复打开多个配置文件的成本。

## 5. 已实现的索引与新鲜度能力

### 5.1 `index_status`

`index_status` 已经不只是“索引存不存在”，而是会返回：

- 索引是否存在
- 索引路径
- 是否是 Git 仓库
- 上次索引时间
- embedding 模型
- embedding 维度
- chunk 数量
- code / kb chunk 数量
- 上次索引 commit
- tracked file 数量
- 索引是否可能过期
- 变更路径数量
- 建议动作
- auto reindex 是否开启
- watcher 是否在跑
- reranker 是否开启
- reranker 模型名

这项能力对客户端的增强在于：

- 模型在做语义检索前可以先判断索引是否可信
- 可以知道当前结果是否可能陈旧
- 可以给出明确的 `reindex auto` 建议，而不是盲猜

### 5.2 `reindex`

当前已支持三种重建模式：

- `full`
- `incremental`
- `auto`

其中 `auto` 会根据环境自动选路：

- 在 Git 仓库中，候选文件来自 `git ls-files --cached --others --exclude-standard`，因此会遵守 `.gitignore`
- 结合 `last_indexed_commit` 与当前文件 manifest 判断变化，已索引过的未提交文件不会反复触发 stale
- 在非 Git 场景中，退化为 `size + sha256` 的 manifest 对比

这项能力的实际价值是：

- 避免每次都全量重建
- 让语义检索尽量跟上项目变化
- 降低本地重建时间和 embedding 计算成本

### 5.3 可选后台自动增量更新

当前已经支持可选 watcher：

- 由 `MCP_SEARCH_AUTO_REINDEX` 控制
- 由 `MCP_SEARCH_AUTO_REINDEX_INTERVAL_SECONDS` 控制轮询间隔

虽然当前还是轮询，不是原生文件事件，但它已经把“索引完全手工维护”推进到了“可自动保持基本新鲜度”的阶段。

## 6. 已实现的 reranker 能力

本项目已经落地 reranker 二阶段重排，而不只是停留在方案设计。

当前已实现特性包括：

- 可选启用 reranker
- 对 `code_semantic_search` 与 `kb_search` 进行二阶段重排
- 支持 `candidate_multiplier`
- 支持 `max_candidates`
- 支持请求超时控制
- 支持内存缓存
- 在结果中保留 `rerank_score`
- reranker 失败时自动回退到原始向量排序

这意味着当前系统已经不是简单的“向量 top-k”，而是“向量粗召回 + 精排筛选”的组合。

对 `Codex` / `Claude Code` 的直接增强有两点：

- 减少误召回导致的错误文件打开
- 提高 top 结果的可用性，降低后续二次探索次数

同时，reranker 缓存也已经具备实际成本价值：

- 减少重复 query / chunk 的公网 rerank 调用
- 降低重复任务的额外网络请求与延迟

## 7. 当前已经实现的降本机制

本项目当前的“降本”不是单点能力，而是多层组合。

### 7.1 搜索与阅读分离

系统已经明确把：

- 搜索
- 结构摘要
- 精确回取

分成不同工具。

这能避免模型直接读取整文件或整目录，是最基础的 token 节约手段。

### 7.2 先便宜判断，再贵价展开

当前推荐流程已经固定为：

- 先 `repo_overview`
- 再 `symbol_search` / `code_exact_search` / `code_semantic_search`
- 最后用 `open_spans` 或 `code_context_pack`

这代表系统把“是否值得展开大上下文”的判断尽量放在本地完成，而不是让远端大模型承担。

### 7.3 精确检索优先于语义检索

系统当前明确区分了：

- 精确查询
- 语义查询

很多真实开发问题其实有明确标识符，此时直接用 `ripgrep` 比 embedding 更省、更快、更准。  
这不仅省 token，也省本地推理成本。

### 7.4 上下文预算控制

当前多个能力已经内置预算控制：

- `open_spans` 限制总字符数
- `code_context_pack` 限制总字符数
- `dependency_overview` 只读取有限的依赖文件和前若干行
- `change_context` 只返回紧凑变更上下文

这意味着“短返回”已经是系统实现层面的默认策略，而不是靠模型自觉克制。

### 7.5 本地上下文压缩

`code_context_pack`、`symbol_context`、`doc_answer_context`、`change_context` 都已经把“搜索后再人工裁剪”的流程搬到了本地。

它们的共同特点是：

- 去重
- 合并
- 限流
- 打包

这类本地预处理正是降低远端模型上下文成本的核心。

### 7.6 reranker 降低误召回成本

reranker 本身不直接减少 token，但它已经在当前实现中起到了明显的间接降本作用：

- 减少把不够相关的片段返回给远端模型
- 减少模型继续打开错误文件的概率
- 提高 top 3 / top 5 的有效性

### 7.7 增量索引降低重建成本

通过 `auto / incremental / full` 三种模式，以及 Git / manifest 差异检测，系统避免了“每次都全量 embedding 全仓”的粗暴做法。

这降低了：

- 本地 embedding 计算成本
- 重建等待时间
- 使用门槛

### 7.8 轻量 symbol 边界切块

当前对 `Python / JS / TS` 已优先按轻量 symbol 边界切块，其他语言再退化为行窗口切块。

这项实现的意义是：

- 尽量让 chunk 更接近真实函数、类、类型边界
- 降低一个 chunk 里混入过多无关代码的概率
- 提高语义检索结果的可读性与可用性

这同样属于间接降本，因为更好的 chunk 通常意味着更少的误命中和更少的二次展开。

## 8. 当前对 Codex / Claude Code 最有价值的实际用法

如果按当前已实现能力来固定工作流，最推荐的用法是：

### 8.1 初次接手项目

- 先用 `repo_overview`
- 再用 `dependency_overview`
- 必要时用 `file_outline`

这样能快速建立项目地图，而不必展开多个大文件。

### 8.2 查具体实现

- 有明确标识符时先用 `code_exact_search`
- 查定义时用 `symbol_search`
- 查相似实现时用 `code_semantic_search`
- 最后用 `open_spans`

这是当前最省 token、最少误判的一条路径。

### 8.3 改某个函数或类

- 先用 `symbol_context`
- 不够再补 `open_spans`

这可以显著减少“先查定义、再查引用、再开文件”的多步开销。

### 8.4 回答项目文档问题

- 先用 `doc_answer_context`
- 不够再补 `kb_search + open_spans`

这比直接读 README、读 docs 更紧凑。

### 8.5 接手本地改动或 review

- 先用 `change_context`
- 再按需要展开 `open_spans`

这比直接通读所有变更文件更适合长会话和中断恢复。

- `callsite_search`

- `session_cache`

这些能力后续可以继续做，但不应写进当前对外固定功能口径。

## 10. 功能冻结结论

如果按“当前已落地能力”来固定产品表述，本项目可以被准确描述为：

这是一个面向 `Codex` / `Claude Code` 的本地 MCP 检索与上下文压缩系统。  
它已经实现代码检索、文档检索、语义召回、reranker 重排、结构摘要、符号聚合、变更上下文、依赖总览、索引新鲜度检查、增量重建以及客户端接入脚本化。

它当前最核心的价值不是“提供更多搜索结果”，而是：

- 让代码助手更快找到对的上下文
- 让代码助手少读无关文件
- 让昂贵模型把 token 花在理解和修改上，而不是花在大范围扫描仓库上

这就是本项目当前已经实现、并且可以稳定写入产品说明的功能边界。
